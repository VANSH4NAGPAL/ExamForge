# Flow Designer & Scripting

**Flow Designer** is considered a natural-language, no-code/low-code tool that enables process owners to automate work. However, developers can absolutely take advantage of powerful scripting capabilities to extend its baseline functionality. 

It has reached feature parity with the legacy **Workflow Editor** and should be used for all new development and major process redesigns in ServiceNow.

---

## Workflow Studio and Flow Designer Overview

The modern way to access Flow Designer is by launching into **Workflow Studio** from your Filter Navigator (or the All menu). When you search for Workflow Studio and click it, it opens in a new tab.

**Workflow Studio** provides a streamlined, centralized place to author, configure, and monitor:
1. **Playbooks**: Design automated processes for your organization (e.g., journey management playbooks for new hire onboarding).
2. **Flows**: An automated process consisting of a single **trigger** and a sequence of **actions** (e.g., a P1 hardware incident triggers certain tasks, updates, and notifications).
3. **Subflows**: A sequence of reusable actions. Subflows do not require a trigger and can be started from a flow, another subflow, or directly from a script. They act similarly to a template for grouping common actions.
4. **Actions**: Singular, reusable operations that automate Now Platform features without having to write code (e.g., sending an email, submitting an approval, updating a record's state, or assigning a group). ServiceNow delivers hundreds of these.
5. **Data Streams**: Create reusable actions to process a stream of response data within a flow.
6. **Decision Tables**: Enables developers to decouple decision logic from their code by creating and maintaining decision rules.
7. **Integrations**: Direct linkage for those working with Integration Hub.

### Flow Designer vs. Workflow Editor
ServiceNow originally utilized Workflow Editor for automating work (e.g., from submitting a record to its closure or fulfillment). Flow Designer is the newer, heavily developed tool. ServiceNow has put tremendous effort into adding additional delivered actions every release (delivering hundreds out-of-the-box compared to very few initially). 

> **Best Practice:** Use Flow Designer for all new implementations and configurations. If you are going to use **Now Assist** capabilities, there are cool AI features available exclusively with Flow Designer that are not available in Workflow Editor. Always evaluate existing flows and actions before starting from scratch.

## Flow Designer Interface and Architecture

### The Home Page
The Flow Designer home page allows you to view flows by type, see a list of workflows you recently updated (the "Pick up where you left off" section), and access helpful resources.

### The Editor Page
When you open a flow, you are presented with the Flow Editor which uses properties, a trigger, and a sequence of actions. You can have multiple flows open in separate tabs.

- **Type Indicator**: An icon indicates if you are working on a flow, subflow, or action.
- **View Toggles**: You can view the flow in the **Default View** (a tree-like structure where you click on an action to expand it downward) or the newer **Diagram View** (which shows the flow side-by-side visually, heavily resembling the old Workflow Editor).
- **Testing and Execution**: You can test your flows directly in the Flow Designer UI before publishing.
- **Three-Dots Menu**: Found in the upper-right corner. It allows you to **Copy** the current object (flow, subflow, or action) or create a **Code Snippet** (crucial for calling subflows/actions via script).
- **Status & Scope**: You can clearly see if a flow is "Published" or "Read Only" and what application scope it belongs to.

### Flow Architecture
A standard Flow consists of:
1. **Trigger**: Specifies the condition that starts the flow.
   - *Record-based*: Runs when a record is created or updated (e.g., `Incident created or updated and is P1`).
   - *Date/Schedule-based*: Runs at a specific time of day, week, or month based on the passage of time.
   - *Application-based*: Runs based on a specific application event. Commonly used for the **Service Catalog**. You can create a generic application trigger and link the flow directly to a specific catalog item in Catalog Builder.
2. **Actions & Flow Logic**: Actions perform the work. **Flow Logic** controls how actions run using statements like `If` or `Else` (e.g., "If first-level approval passes, request second-level approval").
3. **Flow Data (Data Pills)**: Any data gathered or generated by the trigger or previous actions is available in the **Data Panel** on the right side. Each variable is represented by a **Data Pill** that can be dragged and dropped into action input fields to give the flow context (e.g., dropping the `incident record` pill into an "Update Record" action, or dot-walking from that incident to update a related record).
4. **Error Handling**: A dedicated section to catch errors in your flow. If an error occurs, it runs a sequence of actions or subflows to identify/correct the issue.
   - *Example*: Adding an action in the Error Handler to log the error to the system log (`gs.info`, `gs.warn`, `gs.error`) or send an alert email/notification to developers.

## Scripting in Flow Designer

While it is a low-code tool, Flow Designer caters to developers who need custom functionality. Before writing a custom script, always verify that a combination of core delivered action steps cannot achieve the desired result. Also, consider copying an existing action to modify it rather than starting from scratch.

### 1. Inline Scripting
Inline scripting enables developers to write short scripts that set or modify input values during the configuration of an action or flow. 

Instead of dragging a Data Pill, click the **Script Icon** (it looks like `</>` or a small mountain). This opens a Syntax Editor where you can use server-side JavaScript, including classes like `GlideSystem`, `GlideDate`, and `GlideDateTime`.

**Example Use Case (From the Lab):**
You are updating an Incident record's work notes and want to dynamically set a "To be completed by" date to be exactly two days from the opened date.
```javascript
// Access the trigger's opened_at data pill dynamically
var gdt = new GlideDateTime(fd_data.trigger.current.opened_at); 
gdt.addDays(2); // Add two days
return "To be completed by: " + gdt.getDate();
```

### 2. Complex Data Handling
Flow Designer has built-in support for complex structured data (like Arrays and Objects). You can encode and store data in machine-readable formats like **JSON** or **XML**. 
- You can parse and format data without having to write exhaustive parsing code.
- You can create arbitrary data structures and view their organization graphically.
- *Example*: Creating an array of "Contact" objects, where each object contains a `sys_id`, `name`, `email`, and `phone_number`. Flow Designer handles the array iteration cleanly.

### 3. Script-Triggered Subflows
Subflows **do not require a trigger**. They take **Inputs**, process them via **Actions**, and pass back **Outputs**. Subflows can be called entirely from a server-side script (like a Business Rule, Script Include, or UI Action).

**How to call a Subflow from a Script:**
1. Open your published Subflow.
2. Click the **Three-Dots Menu** in the top right.
3. Select **Create Code Snippet**. ServiceNow will automatically generate the boilerplate server-side code needed to pass the required Inputs and execute the subflow.

### 4. Custom Action Script Steps (Action Designer)
If you are building a custom application with custom tables/fields, and no delivered action meets your needs, you can build a **Custom Action**.
When creating an Action, you define **Inputs** (data passed from the flow's data panel into the action) and **Outputs** (data returned back to the flow).

Inside the Action Designer, you can add various steps (like Email, Log, Notification, Payload Builder), but the most powerful is the **Script Step**.
- **Input Variables**: Map the action's inputs to script variables.
- **Script**: Write reusable server-side JavaScript to process the data.
- **Output Variables**: Map the script's results to the action's outputs.

**Example Use Case (From the Lab):**
Creating a script step to calculate the difference between two dates.
- **Inputs**: `sdt` (mapped to `start_date_time`) and `edt` (mapped to `end_date_time`).
- **Script**: Declare two `GlideDateTime` variables using the inputs. Subtract them using the appropriate method (e.g., `GlideDateTime.subtract()`). 
- **Outputs**: Return the `day_difference`.

You can **Test** custom actions directly within the Action Designer by providing sample input values (like a first name, last name, and phone number) and viewing the execution details to debug before publishing.

## Flow Designer APIs

ServiceNow provides specialized APIs to interact with Flow Designer objects programmatically. (Check the developer portal under "Server Scoped" for full details).

### 1. `FlowAPI` (Server-Side)
Used to execute actions, flows, and subflows from any server-side script. 
- Can be executed in a **blocking** (synchronous) or **non-blocking** (asynchronous) mode.
- *Example*: Executing Flow B from inside Flow A via a script.

**Scriptable Flow Runner API Call Order:**
To build and execute flow objects programmatically, use the builder pattern:
1. **Create the build object**: Initiate the scriptable flow runner using `sn_fd.FlowAPI.getRunner()`.
2. **Specify the context to run**: Use methods like `.action()`, `.dataStream()`, `.flow()`, or `.subflow()` to specify what to build. Chain methods like `.addInput()` or `.inDomain()` to pass execution parameters. Finally, chain the `.run()` method (e.g., running a global scope action named `markapproved`).
3. **Retrieve Outputs**: Fetch the outputs from the action executable.

### 2. `FlowScriptAPI` (Inside Flow Designer)
This API allows you to access Flow Designer context details. 
- You **cannot** manually instantiate objects of this type. 
- It is created automatically and is **only accessible inside Flow Designer** script steps and inline scripts.

### 3. `GlideFlow` (Client-Side)
A client-side API library used to start and interact with flows, subflows, and actions directly from the front end (e.g., from a Client Script).
- *Warning*: It is **not immediate**. Because it executes asynchronously, `GlideFlow` returns a JavaScript Promise object representing the eventual result of the operation. There will be a slight delay compared to standard synchronous client-side code.

## Lab Exercises Overview

To solidify these concepts, the course includes three specific labs for Flow Designer:
1. **Build a Flow**: Create a simple flow with a basic trigger and action, and practice writing an **inline script** using the `</>` icon.
2. **Trigger a Subflow with a Script**: Explore the three-dots menu to use the **Code Snippet Generator**, then use that snippet to trigger your subflow from a script.
3. **Add a Script to a Flow**: Use the Action Designer to build a **Custom Action** that utilizes a **Script Step** to process inputs and return an output.
